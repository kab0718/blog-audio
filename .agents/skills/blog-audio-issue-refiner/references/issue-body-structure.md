# Issue本文の構成

次の構成を基準にする。空の節は省略し、Issueに適した分かりやすい見出しへ変更してよい。

```markdown
## 背景・ユーザー課題

誰が、どの状況で、何に困っているか。

## 目的

このIssueで実現するユーザー価値。

## 現状と調査結果

- 現在の挙動
- 関連する実装とファイルパス
- 再利用できる既存パターン
- 制約や既知の問題

## スコープ

### 対象

- このIssueで実装すること

### 対象外

- 今回は実装しないこと

## 要件

- 観測可能で曖昧さのない振る舞い

## 実装方針

- 変更対象と責務
- データや状態の流れ
- 既存設計との整合

## 実装手順

1. 依存順に並べた具体的な変更
2. 各手順で触る主なファイルや境界
3. 検証可能な単位の完了条件

## 受け入れ条件

- [ ] ユーザー視点で確認できる完了条件
- [ ] 主要な異常系や境界条件
- [ ] モバイルUXや音声変換などの関連制約

## 検証方法

- `npm run build`
- 必要な手動確認と対象画面・操作

## 前提・リスク

- 調査に基づく前提
- 未解決事項とその影響
```

## 記述規則

- 元のユーザー意図を本文上部に残す。
- リポジトリのパスはバッククォートで示し、古くなりやすい行番号は付けない。
- 変更対象だけでなく、変更する理由を記載する。
- 受け入れ条件と実装手順を混同しない。
- 未解決のプロダクト判断は`前提・リスク`に記載する。
- 大きなコード断片をIssueへ貼り付けない。
