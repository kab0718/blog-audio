# Blog Audio Issue Refiner

粗い要望または既存のGitHub Issueを調査し、Issue本文を実装可能な計画へ更新する。ユーザーから指定がない限り、Issueは日本語で記述する。

Issueをユーザー意図の一次情報として扱う。`docs/tasks/`は作成しない。タスク文書が必要な場合は`blog-audio-issue-implementation-planner`を使用する。

## ワークフロー

### 1. 初期要望を記録する

- 粗い要望、Issue番号、Issue URLのいずれかを受け取る。
- Issueがない場合は、ユーザーが述べた課題、期待する結果、既知の制約だけを記載して作成する。推測を要件として扱わない。
- 既存Issueの場合は、タイトル、本文、ラベル、関連コメントを読む。
- 最終報告のためIssue番号とURLを保持する。

Issueの作成・編集はリモート状態を変更する。現在の依頼で対象と更新内容が明確に承認されていない限り、書き込み前にリポジトリと変更案を提示して確認する。

### 2. プロジェクトを調査する

最初に次を読む。

- `AGENTS.md`
- `docs/blog_audio_player.md`
- セットアップ、構成、現状動作に関係する場合は`README.md`
- エージェント運用や計画規約に関係する場合は`docs/codex_best_practices.md`

その後、Issueに関係するコード、型、テスト、タスク文書、git履歴だけを調査する。広いディレクトリ読み込みより先に`rg`と`rg --files`を使う。

次を明らかにする。

- 現在のユーザー向け動作と不足
- 影響するモジュール、データフロー、責務境界
- 再利用できる既存パターン
- モバイル再生UX、コードブロック変換、キュー動作、MVP範囲などの制約
- 依存関係、移行、互換性への影響
- `package.json`に実在する検証コマンド
- プロダクト動作や実装範囲を変える曖昧点

ファイル名の列挙だけで終わらせず、変更対象として挙げる根拠を説明できるまで実装を読む。

### 3. スコープと不足情報を解決する

- 1 Issueを1つのまとまった実装単位にする。
- 独立した複数機能が含まれる場合は、本文を書き換える前にIssue分割を提案する。
- 初期要望だけでなく、リポジトリ調査後にも不足情報を確認する。
- ユーザー向け動作、受け入れ条件、スコープ、データや状態、互換性、失敗時動作、実装方針に影響する情報が足りなければユーザーへ質問する。
- 関連する質問を簡潔にまとめ、各回答が必要な理由を示し、回答を得るまで計画を確定しない。
- 重要な不確実性を暗黙の前提にしない。結果を変えない低リスクな実装詳細だけを明示的な前提として扱う。
- ユーザーが判断できない場合は、推奨案とトレードオフを示し、合意した選択をIssueに記録する。
- Issueとリポジトリ資料が矛盾する場合は資料を優先し、矛盾を明記する。
- 明示的に要求されていない低優先度機能を追加しない。

### 4. 更新後のIssue案を作成する

`references/issue-body-structure.md`を使用し、調査した実装に基づく計画を作る。

次を必ず含める。

- ユーザー課題と期待する結果
- 現状と調査根拠
- 対象範囲と対象外
- 検証可能な要件と受け入れ条件
- 具体的な変更対象パスを含む実装方針
- 依存順の実装手順
- 実在するコマンドを使った検証計画
- 前提、リスク、未解決事項

リポジトリに根拠のない実装詳細、不要な抽象化、存在しないコマンドを記載しない。

### 5. セルフレビューして修正する

提示・更新前に、次の観点で完成した案をレビューする。

- **ユーザー価値:** 元の課題を解決し、意図を失っていないか。
- **プロダクト整合性:** `AGENTS.md`、プロダクト資料、モバイル優先、コードブロック読み上げ規則に従っているか。
- **調査根拠:** 現状説明と変更対象が、読んだコードや資料に裏付けられているか。
- **技術的実現性:** 責務境界、状態遷移、依存関係、異常系、実装順が整合しているか。
- **スコープ:** 対象・対象外が明確でMVPに収まっているか。
- **受け入れ条件:** 正常系、重要な異常系、境界条件を確認できるか。
- **検証品質:** 実在するコマンドと適切な手動確認になっているか。
- **Issue品質:** 簡潔で矛盾がなく、追加調査なしで実装を開始できるか。

各指摘は案を修正するか、質問・前提・リスクとして明示する。重大な矛盾、根拠不足、実装を妨げる曖昧さがなくなるまで再レビューする。内部思考は公開せず、重要な判断、修正、未解決事項だけを報告する。

### 6. Issueを更新する

- 主な変更点と完全なタイトル・本文案を提示する。
- 現在の依頼でその案の更新が明示的に承認済みでなければ確認を得る。
- 利用可能なら接続済みGitHubツール、なければ`gh issue edit`を使う。
- 元の有用な文脈、制約、コメント上の決定を失わない。
- ラベル、担当者、マイルストーン、Project項目は明示的な依頼がある場合だけ変更する。

### 7. 更新結果を検証する

- 更新後のIssueをGitHubから再取得する。
- タイトルと本文が承認済みの案と一致することを確認する。
- Issue URL、主要な判断、前提、残る問題を報告する。
- 再取得に成功していない場合は、更新完了と報告しない。

## 品質基準

- 追加の調査なしで次の実装作業が分かる状態にする。
- 変更対象パスを実際の調査根拠と結び付ける。
- 内部実装より先にユーザー向け動作を説明する。
- 受け入れ条件を観測可能にし、実装手順を依存順に並べる。
- GitHub上で読みやすい簡潔さを保つ。
- コードブロックは要約、説明、または読み飛ばしとして扱い、コードをそのまま読み上げない。
- 追加の関連コマンドが定義されていない限り、`npm run build`を主な検証にする。

## 関連スキル

精錬したIssueを別途`docs/tasks/`へ落とす依頼がある場合だけ、後続で`blog-audio-issue-implementation-planner`を使用する。
