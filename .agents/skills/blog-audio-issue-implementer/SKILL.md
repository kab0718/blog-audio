# Blog Audio Issue Implementer

1件の精錬済みGitHub Issueを、専用worktreeで実装からDraft PR作成まで進める。実行単位を分離し、別Issueに対する複数のスキル実行が同時に動いても干渉しないようにする。スキル内部での並列実装は任意であり、必須ではない。

すべての作業を`AGENTS.md`と`docs/blog_audio_player.md`に従わせる。実装中に実際の不足が判明した場合を除き、Issueの計画を書き換えない。

## ワークフロー

### 1. 入力と基準を確認する

- GitHubからIssueのタイトル、本文、受け入れ条件、関連コメントを読む。
- Issueが実装可能な計画を含むことを確認する。動作、スコープ、受け入れ条件に重大な曖昧さがあれば、実装前にユーザーへ質問する。Issue精錬を依頼された場合は`blog-audio-issue-refiner`を使う。
- `AGENTS.md`、`docs/blog_audio_player.md`、Issueで指定された実装箇所を読む。
- `git status --short --branch`、既存worktree、remote、デフォルトブランチを確認する。
- ブランチ作成前に`origin/main`をfetchし、指定がなければ基準にする。
- ユーザーの現在のworktreeにある無関係な変更を含めない。

### 2. 実行専用worktreeを作成する

- `references/invocation-isolation.md`を読む。
- `codex/issue-<番号>-<slug>`形式の一意なブランチと、リポジトリルート配下の`.worktrees/issue-<番号>-<slug>`形式のworktreeパスを決める。
- worktree作成前に、リポジトリローカルの`.git/info/exclude`で`.worktrees/`を除外する。既存内容を保持し、同じ除外設定を重複追加しない。
- 作成前に`git worktree list`とローカルブランチを確認する。他の実行が所有するworktreeやブランチを再利用、reset、削除、上書きしない。
- 同じIssueのworktreeがある場合は、ブランチ、状態、目的が今回の実行と明確に一致するときだけ再開する。それ以外は一意な接尾辞を付けて報告する。
- 更新済みの`origin/main`からブランチとworktreeを作成する。
- worktree作成後、元worktreeの`git status --short`に`.worktrees/`が現れないことを確認する。
- 編集、検証、レビュー修正、コミット、PR公開をすべて専用worktree内で行う。
- ポート、ログ、スクリーンショット、パッチ、一時ファイルをIssueまたは実行単位で分離する。
- ユーザーの元worktreeを変更しない。

### 3. Issueの計画を実装する

編集前に、実装手順を次の観点で整理する。

- 依存関係
- 影響するファイルと責務境界
- 期待する出力
- 検証方法
- 実装順

- 依存順に実装し、編集前に既存パターンを調査する。
- Issueのスコープ内に変更を保ち、既存の責務境界を維持する。
- サブエージェントは、調査や独立レビューに実質的な効果がある場合だけ使う。内部の並列実装や追加worktreeを既定では作らない。
- 計画上の重大な不足、互換性のない依存、スコープ拡大が判明したら停止してユーザーへ確認する。
- 実装中はコミットせず、Issue全体の差分を検証・レビュー完了まで未コミットで保持する。

### 4. 実装を検証する

- 主な検証として`npm run build`を実行する。
- `package.json`に実在するか、Issueで明示されたコマンドだけを追加実行する。
- UI変更では、ブラウザ操作が利用できる場合にアプリを起動し、代表的なモバイル・デスクトップサイズで対象フローを確認する。
- 自動テストがない場合は、受け入れ条件、重要な異常系、境界条件を手動確認する。
- 実行コマンド、結果、検証できなかった制約を正確に記録する。

### 5. 多角的にレビューする

`references/review-checklist.md`を読み、`origin/main`に対するIssue全体の差分をレビューする。

利用可能なら独立したレビュー工程またはサブエージェントを使う。レビュー担当には実装側の結論を渡さず、Issueと生の差分を渡す。少なくとも次を確認する。

- 要件と受け入れ条件への適合
- 正確性、境界条件、回帰リスク
- アーキテクチャ、状態遷移、既存パターンとの整合
- UI変更時のモバイルUXとアクセシビリティ
- コンテンツ・音声変更時の読み上げとコードブロック処理
- 関連するエラー処理、データ整合性、セキュリティ、プライバシー
- 検証内容と不足するテストのリスク
- スコープ遵守とドキュメントの正確性

指摘を重要度順に分類し、パスと行を示す。対応可能な指摘をすべて修正し、影響する検証を再実行して、関連する観点を再レビューする。重大な指摘がなくなるまで繰り返し、公開のためにリスクを隠さない。

### 6. レビュー済みの結果だけをコミットする

- 専用worktreeにIssue対象の変更だけが含まれることを確認する。
- `git diff --check`、`git status --short`、`origin/main`との差分、検証結果を確認する。
- 対象パスを明示的にstageし、事実に沿ったメッセージでコミットする。混在worktreeで`git add -A`を使わない。
- 検証失敗、重大な指摘、未解決のユーザー判断が残る場合はコミット・公開しない。

### 7. pushしてDraft PRを作成する

- 今回の実行ブランチだけを`origin`へpushする。
- 利用可能なら接続済みGitHubツール、なければ`gh pr create --draft`を使い、`main`向けDraft PRを作成する。
- 簡潔なタイトルを付け、本文に次を含める。
  - Issueリンクと、適切な場合はcloseキーワード
  - ユーザー向けの結果
  - 実装概要
  - 実施した検証
  - レビュー範囲と残るリスク
- Ready for reviewへ変更しない。
- PRを再取得し、base、head、Draft状態、タイトル、本文、コミット範囲、URLを確認する。

### 8. 完了を報告する

次を報告する。

- IssueとDraft PRのURL
- ブランチ、worktreeパス、コミットハッシュ
- 検証コマンドと結果
- 実施したレビューと修正内容
- 残る非ブロッキングなリスクや制約

Draft PRが存在し、再取得でDraft状態を確認できるまで完了と報告しない。

## 停止条件

次の場合は公開前に停止してユーザーへ確認する。

- Issueに重要なプロダクト判断が不足している
- Issue外のスコープが必要になった
- 無関係なローカル変更を分離できない
- 検証失敗をスコープ内で修正できない
- 重大なレビュー指摘が残っている
- push先またはPR作成先が曖昧である
- 対象worktreeまたはブランチを別の実行が使用している

## 関連スキル

- Issue自体の精錬が必要なら`blog-audio-issue-refiner`を使用する。
- 別途`docs/tasks/`の計画を求められた場合だけ`blog-audio-issue-implementation-planner`を使用する。
- 同じ公開処理に`blog-audio-git-publish`を併用しない。このスキルがIssueブランチ、コミット、push、Draft PRを担当する。
