# 実装レビュー・チェックリスト

## 要件

- 対象要件がコードと検証手順に対応している。
- すべての受け入れ条件を確認できる。
- 元のユーザー意図が維持されている。
- 未承認の機能やリファクタリングが差分に含まれていない。

## 正確性

- 関連する正常、空、読み込み、失敗、再試行、境界状態が一貫して動作する。
- 状態遷移と永続化が整合している。
- 非同期処理が、古い応答、キャンセル、コンポーネントのライフサイクルを適切に扱う。
- 型定義と実行時の前提が一致する。

## アーキテクチャ

- 記事取得、抽出、ナレーション、音声生成、再生、キューの責務境界を守っている。
- 適切な既存パターンと抽象化を再利用している。
- 共有契約の所有箇所が明確である。
- 新しい依存や外部サービスに明確な必要性がある。

## プロダクトとUX

- スマートフォンでの再生体験を最優先している。
- モバイルサイズで操作しやすく、意味が明確で、レイアウトが安定している。
- 読み込み、エラー、無効、空状態が利用可能である。
- 関連するキーボード、フォーカス、ラベル、コントラスト、動きの軽減を確認している。
- 技術記事のコードブロックを逐語読みせず、要約、説明、または読み飛ばしとして扱っている。

## 信頼性と安全性

- エラー時に再生、キュー、記事状態を破損しない。
- ユーザー入力URLと外部コンテンツを防御的に扱う。
- ログやUIに秘密情報や不要な個人情報を出さない。
- キャッシュと永続化に互換性のない古いデータを残さない。

## 検証

- `npm run build`が成功する。
- 関連する既存チェックが成功し、存在しないコマンドを実行済みと主張していない。
- 手動確認が受け入れ条件と回帰リスクの高いフローをカバーしている。
- スコープ内で追加できない自動テスト不足を残存リスクとして記録している。

## 公開ゲート

次が残る場合はコミットとDraft PR作成を止める。

- 未達の受け入れ条件
- 必須検証の失敗
- 未解決の重大な正確性・セキュリティ問題
- ユーザー判断が必要な重要な曖昧点
- 説明できない、または無関係なファイル変更
- PR差分とレビュー済みworktree差分の不一致
