# アプリ骨格とモバイル向け画面遷移を作る

## 1. Task Summary

- Goal: モバイルファーストの Web アプリ土台を作り、記事一覧・プレーヤー・キューの 3 画面を同じ骨格上で遷移できるようにする
- User value: スマホで記事選択から再生、キュー確認までの基本導線を自然に辿れる状態を先に固められる
- Related priorities: smartphone playback UX, continuous playback / queue behavior, simple MVP scope
- Source docs:
  - `AGENTS.md`
  - `docs/blog_audio_player.md`
  - Issue #1 `MVP: アプリ骨格を作る` <https://github.com/kab0718/blog-audio/issues/1>

## 2. Scope

### In scope

- 初期 Web アプリ基盤の採用とディレクトリ構成の確定
- 記事一覧、プレーヤー、キューの 3 画面へ遷移できるルーティングまたは画面切り替えの実装
- スマホ幅を前提にした共通レイアウト、ヘッダー、ナビゲーションの最小構成
- 画面共通で表示できるミニプレーヤーまたは再生状態表示スロットの配置
- 後続タスクが差し込める仮データ注入ポイントと共通状態の配置場所の定義

### Out of scope

- 実データ取得や記事ソース連携
- 音声生成、音声再生、シークなどの実再生ロジック
- 詳細なビジュアル磨き込みやブランド表現
- PWA 機能、オフライン対応、通知などの追加プラットフォーム対応

### Assumptions

- 初期ターゲットはモバイル向け Web アプリで、PWA 対応は後続判断とする
- 記事一覧と再生状態は当面モックデータでよい
- 共通状態は「置き場所と責務」を先に決め、再生ロジック本体は後続タスクで実装する
- デスクトップ最適化は崩れない範囲に留め、設計基準は iPhone 幅相当を優先する

## 3. UX / Behavior

### Primary flow

- 初回表示で記事一覧画面に入り、再生候補を見つけられる
- 記事一覧から 1 タップでプレーヤー画面へ移動できる
- どの画面からでも 1 タップでキュー画面へ移動できる
- 再生中は画面下部の共通領域からプレーヤーへ戻れる構造にする

### Important states / edge cases

- iPhone 幅相当でも横スクロールが発生しない
- 画面切り替え時に固定下部領域がレイアウト崩れを起こさない
- コンテンツ量が少ない画面でも下部ナビゲーションと再生状態表示が重ならない
- 再生状態が未接続でも、後続実装で差し替え可能なプレースホルダー表示を維持する

## 4. Requirements

### Functional requirements

- 記事一覧、プレーヤー、キューの 3 画面を明示的に切り替えられる
- 3 画面は共通のアプリシェル上に表示される
- 共通レイアウト上に、再生状態やミニプレーヤーを後から実装できる固定領域がある
- 仮データで各画面の見出し、主要領域、ナビゲーション導線を確認できる
- 後続タスクが記事一覧 UI、プレーヤー UI、キュー UI を独立して実装できる責務分離がある

### Non-functional constraints

- スマホ幅を第一基準にし、主要操作は親指で届く下部寄りの導線を優先する
- 1 画面 1 責務を崩さず、一覧・再生・キューの役割を混在させない
- 実データや再生ロジックに強く結合しない骨格にする
- 後続タスクで UI 差し替えや状態管理方式の変更があっても破綻しにくい構造にする

## 5. Technical Approach

### Proposed approach

- `Vite + React + TypeScript` をアプリ基盤に採用し、`app shell`、`screen modules`、`shared state` の責務を分離する
- 画面遷移は `React Router` の URL ベースで実装し、一覧 / プレーヤー / キューを明示的な画面として扱う
- 共通レイアウトで `header`、`main content`、`bottom navigation`、`mini player slot` の 4 領域を定義する
- 記事一覧、プレーヤー、キューは仮データ表示で先に繋ぎ、見た目よりも導線と構造を優先する

### Data / API / state considerations

- 記事一覧用のモック配列と、現在再生中の記事 ID やキュー件数などの最小状態を用意する
- 再生状態は React Context + `useReducer` で共通配置し、ロジック未実装時はダミー値で代替する
- 画面コンポーネントはデータ取得を内包せず、外から注入されたデータで描画できるようにする
- スタイルは `CSS Modules + CSS variables` を基本にし、画面単位の見通しとテーマ調整のしやすさを両立する

### Dependencies

- 先にこのタスクでアプリ土台を固定しておくと、後続の一覧 UI、プレーヤー UI、キュー UI を並行実装しやすい
- 実再生ロジックや記事取得はこの骨格の上に積む前提にする

## 6. Risks / Open Questions

- Web アプリ基盤は `Vite + React + TypeScript` を前提とする
- 画面遷移は `React Router` の URL ベースを採用するが、URL 設計をどこまで MVP で固定するかは今後の画面追加次第
- 共通状態は React Context + `useReducer` で開始するが、再生キューや再生制御が肥大化した場合の分割方針は別途整理が必要
- スタイルは `CSS Modules + CSS variables` を採用するが、デザイントークンの粒度は初期実装で定める必要がある
- ミニプレーヤー領域はこのタスクでは固定表示の枠のみを実装し、中身のダミー状態表示は後続タスクに委ねる
- PWA 対応は初回スコープ外とし、ホーム画面追加やブラウザ UI の占有領域は別タスクで実機確認する

## 7. Acceptance Criteria

- スマホ幅で記事一覧、プレーヤー、キューの各画面へ遷移できる
- どの画面でも共通の下部固定領域を保持し、後続の再生 UI を載せられる
- 主要導線が横スクロールなしで操作できる
- 各画面がモックデータで独立表示され、後続タスクが UI 実装を着手できる
- 実データ取得や音声再生ロジックを未実装のままでも、アプリ骨格として破綻しない

## 8. Implementation Plan

1. `Vite + React + TypeScript` を前提に初期プロジェクト構成とディレクトリ構成を作る
2. `React Router` を使って記事一覧、プレーヤー、キューの 3 画面を切り替えられる土台を作る
3. アプリ全体の共通レイアウトを実装し、ヘッダー、メイン領域、下部ナビゲーション、ミニプレーヤー用の固定スロットを配置する
4. 記事一覧、プレーヤー、キューの 3 画面を仮データで描画し、スマホ幅で破綻しないことを確認する
5. React Context + `useReducer` による共通状態の配置場所だけを整え、後続の再生 UI 実装を差し込める状態にする

## 9. Validation

- Available commands:
  - `npm install`
  - `npm run dev`
  - `npm run build`
  - `npm run preview`
- Manual checks:
  - スマホ幅で記事一覧、プレーヤー、キューを往復し、横スクロールや重なりがないことを確認する
  - 下部固定領域が各画面で表示され続け、主要コンテンツを覆わないことを確認する
  - 仮データの差し替え前提で、各画面が独立して描画されることを確認する
- Known limitations:
  - 実再生ロジックや記事取得が未実装のため、導線とレイアウト中心の検証に留まる
  - PWA 挙動や実機ブラウザ固有差分は別タスクでの検証が必要
